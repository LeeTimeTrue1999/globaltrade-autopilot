# 跨境电商运转方式与当前框架问题清单

## 1. 现实中的跨境电商怎么运转

跨境电商不是单一“选品 + 上架”动作，而是多条链路叠在一起：

```text
市场研究
  -> 账号/主体/平台准入
  -> 选品与合规筛查
  -> 国内供应商匹配
  -> 成本、税费、物流和汇率测算
  -> 上架内容本地化
  -> 库存与履约方式选择
  -> 流量投放/达人/活动
  -> 订单处理
  -> 售后、退款、差评和退货
  -> 平台结算、换汇和财务对账
  -> 复盘、补货、调价、下架
```

成熟卖家通常不是靠一个热品表运转，而是围绕以下能力闭环：

- 多平台商品、订单、库存和结算数据同步。
- 平台规则、目的国合规和禁售规则维护。
- 多供应商、多物流、多收款账户备份。
- 毛利、现金流、库存周转和售后损耗复盘。
- 内容/广告/达人驱动的流量管理。
- 风险事件处理：延迟发货、断货、退款、平台处罚、货损、汇率波动。

## 2. 主要业务模式

### 2.1 平台型经营

代表：Shopee、TikTok Shop、Ozon、Amazon、Lazada、Wildberries。

特点：

- 流量、交易、买家支付主要由平台掌控。
- 卖家负责商品、价格、库存、履约、售后和合规。
- 平台会抽佣、收取支付/物流/广告/仓储等费用。
- 平台规则变化会直接影响利润和订单状态。

对系统的含义：

- 必须按平台保存订单 ID、商品 ID、SKU、结算批次、费用项。
- 平台后台导出和官方 API 都要作为一等数据源。
- 不能只算“销售价 - 采购价”，必须算平台真实扣费和退款损耗。

### 2.2 内容电商/达人分销

代表：TikTok Shop。

特点：

- 商品机会常来自内容热度、短视频、直播和达人带货。
- 销量可能暴涨，但也会带来库存和履约压力。
- 达人佣金、样品、优惠券、直播坑位和内容素材都进入成本。

对系统的含义：

- 只靠商品维度不够，需要新增 `campaign`、`creator`、`content`、`affiliate_commission`。
- 评分模型要区分“自然需求”和“内容驱动需求”。
- 需要记录视频/直播带来的订单归因，否则复盘会失真。

### 2.3 履约模式

常见模式：

| 模式 | 说明 | 优点 | 风险 |
| --- | --- | --- | --- |
| 平台仓 / FBO / FBA / FBT | 货先进平台仓，由平台仓储、拣货、配送、退货 | 时效和转化更好 | 压库存、仓储费、滞销、货损、仓内异常 |
| 卖家自发货 / FBS / FBM | 卖家自己打包发到平台/物流商 | 库存控制强 | 时效、异常、平台罚分压力大 |
| 一件代发 / Dropship | 供应商或仓库按单发货 | 现金压力低 | 质量、时效、断货和售后不可控 |
| 海外仓 | 提前备货到目标市场 | 时效好、可扩量 | 资金占用、清关和库存周转风险 |

对系统的含义：

- 当前 `supportsDropship` 太粗，需要建模履约模式、备货地点、库存责任、退货责任。
- 利润测算要把仓储费、入库费、长期仓储、退货处理和库存报废纳入。
- 小单测试和放量阶段应使用不同履约策略。

### 2.4 支付与结算

真实资金链一般分成：

- 买家支付给平台。
- 平台扣除佣金、支付费、退款、广告费、物流费后结算给卖家。
- 卖家通过跨境收款账户、银行卡或第三方收款服务提现。
- 卖家向国内供应商、物流商、服务商付款。
- 财务按订单、结算批次、采购单和付款流水对账。

对系统的含义：

- 支付状态不等于结算到账。
- 订单利润不能只在下单时预估，必须在结算后形成实际利润快照。
- 支付宝 MCP 适合国内采购付款和对账，不适合替代平台买家支付链路。

## 3. 当前框架下的主要问题

### P0：数据导入和来源追踪还没成为核心模块

当前文档已经强调数据源，但产品功能还没有把“数据来源、置信度、导入批次、原始文件、字段映射”放进核心流程。

问题：

- 机会分可能混合官方订单、人工观察和估算数据。
- 没有数据 lineage 时，推荐动作很难被信任。
- 后续接 API 时会遇到字段口径和人工导入口径不一致。

建议：

- M1 第一优先做 `Import Center`。
- 所有市场商品、供应商、物流报价、结算数据都先进入 staging。
- 每条数据保存 `source_type`、`source_file_or_api`、`confidence_level`、`import_batch_id`。

### P0：缺少库存和资金占用视角

当前模型重视机会分和毛利，但跨境真正容易出问题的是库存周转和现金流。

问题：

- “小单测试 5-20 单”还没有绑定采购量、库存量、安全库存和现金占用。
- 没有库存老化、断货、滞销、补货周期、在途库存。
- 毛利高但周转慢的商品可能拖垮现金流。

建议：

- 新增 `InventorySnapshot`、`StockMovement`、`CashflowSnapshot`。
- 机会评分加入 `InventoryRiskScore` 或 `CashCycleDays`。
- 周报加入库存周转、滞销天数、缺货风险、在途金额。

### P0：履约模型过于简化

当前供应商模型只有 MOQ、发货天数、是否一件代发，不能覆盖平台仓、自发货、海外仓和俄罗斯特殊线路。

问题：

- 同一个商品在不同国家/平台可能需要不同履约方式。
- 国际物流成本不是固定字段，应按重量、体积、线路、敏感货、时效计算。
- 退货责任和异常责任没有建模。

建议：

- 新增 `FulfillmentPlan`、`Warehouse`、`LogisticsRoute`、`ReturnPolicy`。
- 机会分中区分“采购履约”和“国际履约”。
- 俄罗斯方向单独配置线路、清关、支付和制裁检查。

### P1：平台规则和合规规则还不是版本化资产

当前已有风险规则，但更像静态清单。

问题：

- 平台禁售规则、类目佣金、发货时限、处罚规则会变。
- 目的国法规和认证要求按国家/类目差异很大。
- 没有版本和生效日期，会导致历史利润和历史审核无法解释。

建议：

- 建立 `RuleSet`、`RuleVersion`、`PolicySource`。
- 所有规则有 `effective_from`、`source_url`、`reviewed_by`。
- 上架审核引用当时规则版本，不能只引用最新规则。

### P1：内容电商能力不足

TikTok Shop 不只是平台上架，它强依赖内容和达人。

问题：

- 当前没有达人、佣金、样品、短视频、直播、活动数据模型。
- 广告成本率固定估算，无法分析内容带来的真实转化。
- “销量线索”没有区分自然搜索、推荐流、达人带货和活动补贴。

建议：

- 新增 `MarketingCampaign`、`CreatorCollaboration`、`ContentAsset`、`AffiliateCommission`。
- 报表按渠道拆分 GMV、订单、毛利和退款。
- 对 TikTok Shop 的机会评分增加内容信号，但降低其置信度权重。

### P1：结算和实际利润闭环还不够

已有支付结算设计，但主流程还没把结算回填进机会和订单复盘。

问题：

- 预估毛利和实际毛利没有分层。
- 平台结算批次、退款、广告扣费、物流补扣没有进入订单级对账。
- 汇率快照和实际到账汇率可能不同。

建议：

- 区分 `EstimatedProfitSnapshot` 和 `ActualProfitSnapshot`。
- 订单、结算批次、采购付款、物流账单、退款记录做对账状态。
- 周报必须展示“预估毛利 vs 实际毛利偏差”。

### P1：多平台 SKU 映射会成为早期痛点

当前数据模型有 SKU，但没有 SKU 映射体系。

问题：

- 同一国内商品可能对应多个平台、国家、语言标题和规格。
- 平台 SKU、内部 SKU、供应商 SKU、采购 SKU 容易混乱。
- 订单履约和利润核算需要稳定 SKU 关系。

建议：

- 新增 `ProductMaster`、`Variant`、`PlatformListing`、`SupplierSkuMapping`。
- 所有订单先映射到内部 SKU，再计算履约和利润。

### P2：退货、差评和售后没有进入选品反馈

跨境商品是否能长期做，不只看首单毛利。

问题：

- 差评原因、退货原因、破损率、物流时效没有反馈到商品评分。
- 低评分高销量是机会，但也可能是高售后风险。

建议：

- 新增 `ReviewInsight`、`ReturnReason`、`AfterSalesCase`。
- 机会评分加入售后风险，周报展示退款率和差评关键词。

### P2：账号、主体、店铺维度不足

平台经营不是只有平台名，还涉及店铺、主体、站点、收款账户、授权状态。

问题：

- 当前 `platform` + `country` 不足以支撑真实 API 和结算。
- 多店铺、多主体、多收款账户时会混乱。

建议：

- 新增 `BusinessEntity`、`PlatformAccount`、`Shop`、`SettlementAccount`。
- API Token、结算、订单、商品都绑定到店铺和主体。

## 4. 对当前路线图的调整建议

### M1 必须前置

- 数据导入中心。
- 数据来源和置信度。
- 内部 SKU 映射。
- 供应商和物流报价模板。
- 预估利润快照。

### M2 再做

- PostgreSQL 持久化。
- 用户、权限、审批、审计。
- 库存和现金流。
- 订单导入和结算导入。
- 实际利润对账。

### M3 再接 API

- Shopee、TikTok Shop、Ozon 商品/订单/结算 API。
- 物流轨迹 API。
- 汇率同步。
- 平台规则版本化。

### 暂缓

- 自动发布。
- 自动付款。
- 自动调价执行。
- AI 自动决定选品。

这些可以先做“建议”和“草稿”，但不要自动执行。
