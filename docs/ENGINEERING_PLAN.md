# 完整工程方案

## 1. 当前结论

这个系统建议分三种运行形态推进：

| 阶段 | 使用场景 | 数据库 | 部署方式 | 适合人数 |
| --- | --- | --- | --- | --- |
| 本地原型 | 验证界面、流程、评分规则 | 无数据库，使用本地样例数据 | 本机静态服务 | 1 人 |
| 内部 MVP | 真实录入商品、供应商、订单和审核 | PostgreSQL，优先本地服务器或云托管 | 单台服务器 + Docker | 2-10 人 |
| 正式生产 | 多人长期使用、接平台 API、保存订单和敏感数据 | 云托管 PostgreSQL 或独立数据库服务器 | 应用服务器 + 数据库 + 对象存储 + 备份 | 10 人以上 |

推荐路线：

1. 先保留当前静态原型，继续完善前端交互。
2. MVP 阶段使用 `PostgreSQL`，不要用纯文件或浏览器 localStorage 存业务数据。
3. 如果只是你一个人试跑，可以先用本机或内网服务器数据库。
4. 如果要多人共享，建议直接上服务器，并配置登录、权限、HTTPS、备份和审计日志。
5. 如果会接入真实平台 API 和订单数据，建议用云托管数据库，降低备份、恢复和安全维护压力。

## 2. 技术栈建议

### MVP 推荐栈

| 层 | 推荐 | 原因 |
| --- | --- | --- |
| 前端 | Next.js / React | 适合内部工具、表格、表单、权限页面 |
| 后端 | FastAPI 或 NestJS | API 清晰，易维护，适合任务队列和平台集成 |
| 数据库 | PostgreSQL | 结构化业务数据、事务、查询、报表都稳 |
| 缓存/任务 | Redis | 定时同步、队列、重试、缓存 |
| 文件存储 | S3 兼容对象存储 / MinIO | 商品图片、导入文件、导出报表 |
| 部署 | Docker Compose 起步 | 简单、可迁移、便于服务器部署 |
| 反向代理 | Nginx / Caddy | HTTPS、域名、静态资源、转发 API |

### 为什么不用 SQLite 作为正式库

SQLite 可以用于本地开发或极早期 Demo，但不适合这个系统长期使用：

- 多人同时操作能力有限。
- 权限、审计、报表、并发订单处理会逐渐吃力。
- 后续迁移到 PostgreSQL 会增加成本。

所以建议：

- 本地开发可以用 PostgreSQL Docker。
- 正式 MVP 直接用 PostgreSQL。

## 3. 数据库选择：线上还是线下

### 方案 A：本机数据库

适合：

- 你自己试用。
- 不保存真实订单和 API Key。
- 只做流程验证。

优点：

- 成本低。
- 不需要服务器。
- 数据不出本机。

缺点：

- 不能方便共享给别人。
- 本机坏了数据可能丢。
- 不适合接平台 API。
- 不适合长期运行定时任务。

结论：

- 只适合原型和个人试跑，不建议作为 MVP 正式方案。

### 方案 B：公司内网服务器数据库

适合：

- 小团队内部使用。
- 团队在同一个办公网络或 VPN。
- 对数据外放比较敏感。

优点：

- 数据在自己控制的机器上。
- 成本可控。
- 访问范围容易限制在内网。

缺点：

- 需要自己做备份。
- 需要维护系统安全更新。
- 外出访问需要 VPN。
- 服务器故障需要自己处理。

结论：

- 如果你有技术人员维护服务器，这是一个不错的 MVP 方案。

### 方案 C：云服务器自建 PostgreSQL

适合：

- 要远程共享给团队。
- 想成本可控。
- 还没到高可用阶段。

优点：

- 一台服务器即可跑应用、数据库、Redis。
- 部署简单。
- 团队可通过域名访问。

缺点：

- 数据库和应用在同一台机器，故障影响大。
- 备份、恢复、安全加固需要自己做。
- 数据库暴露风险要严格控制。

结论：

- 适合 MVP 早期，但要配置每日备份。

### 方案 D：云托管 PostgreSQL

适合：

- 多人正式使用。
- 保存订单、供应商、平台 API、财务数据。
- 后续接入平台 API 和自动任务。

优点：

- 自动备份。
- 恢复更方便。
- 数据库不必暴露公网。
- 扩容和监控更成熟。

缺点：

- 成本更高。
- 需要配置云厂商网络和权限。

结论：

- 正式生产推荐这个方案。应用服务器可以坏，数据库不能随便丢。

## 4. 推荐部署阶段

### 阶段 0：当前本地原型

已完成：

- 静态前端。
- 本地评分逻辑。
- 示例数据。
- 零外部依赖。

运行：

```bash
npm run dev
```

用途：

- 确认页面布局。
- 确认业务流程。
- 确认评分规则。

### 阶段 1：本地可持久化 MVP

目标：

- 在本机跑完整前后端。
- 数据保存到本机 PostgreSQL。
- 支持真实录入和导入导出。

需要准备：

- Node.js。
- Docker Desktop 或本地 PostgreSQL。
- 本地 `.env.local`。

服务：

```text
web: 前端
api: 后端
postgres: 数据库
redis: 队列和缓存
minio: 本地对象存储，可选
```

适合：

- 你自己连续使用。
- 小范围验证真实数据。

### 阶段 2：单台服务器内部版

目标：

- 给 2-10 人共享使用。
- 有登录、权限、审计。
- 每日自动备份。

服务器最低建议：

```text
CPU: 2 核
内存: 4 GB
磁盘: 80 GB SSD
系统: Ubuntu LTS / Debian
网络: 固定公网 IP 或公司内网 IP
```

更稳妥配置：

```text
CPU: 4 核
内存: 8 GB
磁盘: 160 GB SSD
```

需要准备：

- 域名或固定 IP。
- HTTPS 证书。
- Docker 和 Docker Compose。
- 服务器防火墙。
- 数据库备份目录。
- 管理员账号。
- 邮箱或企业微信/飞书通知，可选。

端口建议：

```text
80/443: Web 访问
22: SSH，仅允许管理员 IP
5432: PostgreSQL，不开放公网
6379: Redis，不开放公网
9000: 对象存储，不开放公网或仅内网
```

### 阶段 3：正式生产版

目标：

- 接真实平台 API。
- 保存订单、客户信息、平台授权、财务数据。
- 支持长期稳定运行。

推荐结构：

```text
Web/API 应用服务器
  |
私有网络
  |
托管 PostgreSQL
Redis
对象存储
日志/监控
备份系统
```

需要准备：

- 云服务器。
- 云托管 PostgreSQL。
- Redis。
- 对象存储 Bucket。
- 域名和 HTTPS。
- 密钥管理方案。
- 日志和监控。
- 备份和恢复演练。

## 5. 多人共享需要什么

如果要给别人一起用，至少需要：

### 5.1 账号和权限

必须实现：

- 登录。
- 用户管理。
- 角色权限。
- 禁用用户。
- 密码重置。
- 操作审计。

建议角色：

- 管理员。
- 运营负责人。
- 选品运营。
- 供应链运营。
- 店铺运营。
- 财务只读。

### 5.2 数据隔离和操作边界

内部工具也要做边界：

- 选品运营可以创建机会，但不能发布商品。
- 店铺运营可以处理上架草稿，但不能改采购成本。
- 供应链运营可以改供应商和采购价，但不能调平台售价。
- 财务只能看利润和报表。
- 高风险商品必须合规审核。

### 5.3 审计日志

多人使用后，审计日志必须上线：

- 谁改了价格。
- 谁批准了上架。
- 谁导出了订单。
- 谁修改了供应商。
- 谁改了评分规则。
- 谁操作了平台授权。

### 5.4 访问方式

可选方式：

| 方式 | 适合 | 说明 |
| --- | --- | --- |
| 公司内网访问 | 团队都在办公室 | 最安全，但远程不方便 |
| VPN 访问 | 分布式团队 | 安全性好，需要配置 VPN |
| 公网域名 + 登录 | 远程团队 | 最方便，但安全要求最高 |
| 白名单 IP | 固定办公地点 | 简单有效，但移动办公不方便 |

推荐：

- MVP：公网域名 + HTTPS + 登录 + 强密码 + 管理员白名单。
- 正式版：VPN 或零信任访问 + HTTPS + 角色权限。

## 6. 服务器需要额外准备什么

### 6.1 基础资源

- 一台 Linux 服务器。
- 域名。
- HTTPS 证书。
- Docker / Docker Compose。
- 数据盘或独立磁盘目录。
- 防火墙规则。

### 6.2 环境变量

至少需要：

```text
DATABASE_URL=
REDIS_URL=
JWT_SECRET=
ENCRYPTION_KEY=
APP_BASE_URL=
OBJECT_STORAGE_ENDPOINT=
OBJECT_STORAGE_ACCESS_KEY=
OBJECT_STORAGE_SECRET_KEY=
```

接平台后还需要：

```text
SHOPEE_CLIENT_ID=
SHOPEE_CLIENT_SECRET=
TIKTOK_SHOP_CLIENT_ID=
TIKTOK_SHOP_CLIENT_SECRET=
OZON_CLIENT_ID=
OZON_API_KEY=
```

注意：

- 这些密钥不能进前端。
- 不能提交到代码仓库。
- 不能写入日志。

### 6.3 备份

最低要求：

- PostgreSQL 每日备份。
- 保留最近 7 天。
- 每周保留 1 份，至少保留 4 周。
- 对象存储文件定期备份。
- 每月至少做一次恢复测试。

备份文件建议：

```text
backup/
  postgres/
    daily/
    weekly/
  object-storage/
```

### 6.4 监控

MVP 最少监控：

- 服务是否在线。
- 磁盘是否快满。
- 数据库连接是否正常。
- 定时任务是否失败。
- API 同步是否失败。

正式版增加：

- 错误日志。
- 慢查询。
- 任务队列积压。
- 登录异常。
- 导出异常。

## 8. API 和后台任务规划

### API

第一阶段：

- 商品机会 CRUD。
- 供应商 CRUD。
- 上架草稿 CRUD。
- 订单导入。
- 配置管理。
- 登录权限。

第二阶段：

- 平台授权。
- 平台商品同步。
- 订单同步。
- 物流同步。
- 策略动作执行。

### 后台任务

需要任务队列：

- 每日刷新汇率。
- 每日生成机会评分。
- 每日生成经营日报。
- 每周生成经营周报。
- 平台订单同步。
- 上架状态同步。
- 失败任务重试。

任务必须可见：

- 任务名称。
- 开始时间。
- 结束时间。
- 成功/失败。
- 失败原因。
- 重试次数。

## 9. 安全基线

上线给多人使用前必须完成：

- HTTPS。
- 登录认证。
- 强密码或企业登录。
- 角色权限。
- 审计日志。
- API Key 加密。
- 数据库不开放公网。
- Redis 不开放公网。
- 服务器防火墙。
- 自动备份。
- 管理员账号最小化。

不能上线的情况：

- 没有登录。
- 数据库暴露公网。
- API Key 写在前端。
- 没有备份。
- 所有人都是管理员。
- 真实订单导出没有审计。

## 10. 成本和复杂度建议

### 个人试跑

成本最低：

- 本机运行。
- 不接真实 API。
- 手动导入 CSV。

### 小团队 MVP

推荐：

- 单台云服务器。
- Docker Compose。
- PostgreSQL + Redis 同机。
- 每日备份。

### 正式运营

推荐：

- 应用服务器。
- 托管 PostgreSQL。
- 托管对象存储。
- 独立 Redis。
- 监控和告警。
- VPN 或零信任访问。
