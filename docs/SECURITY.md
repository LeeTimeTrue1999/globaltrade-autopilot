# 安全与合规设计

## 1. 基本边界

当前项目按内部工具设计，不对公网开放。

MVP 原型：

- 不联网。
- 不安装依赖。
- 不读取工作区外文件。
- 不保存真实 API Key。
- 不执行平台发布、采购或付款动作。

## 2. 访问控制

正式版本需要：

- 登录认证。
- 基于角色的权限控制。
- 关键动作二次确认。
- 高风险动作审批。
- 审计日志。

权限原则：

- 财务和老板只读经营报表。
- 选品运营不能发布商品。
- 供应链运营不能改平台售价。
- 店铺运营不能修改采购成本模板。
- 系统管理员不能绕过业务审核直接发布。

## 3. 密钥管理

平台 API Key、物流 API Key、汇率服务 Key 必须：

- 加密存储。
- 不进入前端。
- 不写入日志。
- 不提交到仓库。
- 支持轮换。

开发环境使用 `.env.local`，生产环境使用密钥管理服务。

## 4. 外部数据安全

优先级：

1. 官方开放平台 API。
2. 平台导出的 CSV/Excel。
3. 合规第三方数据供应商。

禁止：

- 安装不明浏览器插件。
- 运行来源不明的自动化脚本。
- 绕过平台反爬和访问限制。
- 导入不可信可执行文件。
- 使用来历不明的 cracked 工具。

## 5. 商品合规

默认高风险类目：

- 食品。
- 保健品。
- 药品。
- 医疗器械。
- 化妆品功效品。
- 儿童用品。
- 电池和充电宝。
- 无线通信设备。
- 品牌/IP 商品。
- 军警、双用途或受限制物品。

高风险类目默认不允许自动发布，必须人工合规复核。

## 6. 俄罗斯专项风控

俄罗斯市场需要单独处理：

- 支付和结算限制。
- 汇兑风险。
- 制裁限制。
- 禁运和受限商品。
- 物流线路稳定性。
- 平台主体要求。

系统中俄罗斯商品的默认风险基础分应高于东南亚市场。

## 7. 审计日志

记录内容：

- 操作人。
- 操作时间。
- 操作对象。
- 操作前后差异。
- IP/设备信息。
- 审批链路。

审计日志不可被普通管理员删除。

## 8. 数据保护

订单和买家信息属于敏感信息：

- 最小化展示。
- 加密存储。
- 导出需要权限。
- 导出操作写审计日志。
- 超期数据归档或脱敏。
