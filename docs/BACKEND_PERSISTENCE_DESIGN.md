# Backend Persistence Design

This document defines the first backend persistence layer needed to move GlobalTrade Autopilot from a static frontend prototype to a multi-user internal business system.

Scope:

1. `import_batches + field_mappings + original file object storage`
2. `supplier_matches + logistics_rate_cards`
3. `audit_logs + users/roles`

The design assumes PostgreSQL as the primary database, Redis for jobs/cache, and S3-compatible object storage for uploaded files.

## Design Principles

- Keep raw imported files. Parsed rows alone are not enough for audit, rollback, or dispute resolution.
- Separate staging from committed business records.
- Never overwrite historical financial inputs such as supplier price, FX rate, cost template, or logistics quote without keeping a version.
- Every management-side change must create an audit log.
- Operations users can act on existing data; management/admin users maintain mappings, suppliers, logistics, risk, cost, and permissions.
- IDs should be UUIDs in the backend even if the static prototype uses readable sample IDs.

## 1. Import Batches, Field Mappings, And Original Files

### Business Flow

```text
Upload file
  -> store original file in object storage
  -> create import_batch
  -> parse rows into import_batch_rows
  -> infer/apply field_mapping
  -> validate rows
  -> preview errors/warnings
  -> user confirms import
  -> write normalized business tables
  -> write audit_log
```

### Object Storage Layout

Use one private bucket for application files.

```text
globaltrade-files/
  imports/
    {yyyy}/{mm}/{import_batch_id}/original/{filename}
    {yyyy}/{mm}/{import_batch_id}/normalized-preview.json
    {yyyy}/{mm}/{import_batch_id}/validation-report.json
  exports/
    {yyyy}/{mm}/{export_id}/{filename}
  product-images/
    {market_product_id}/...
  backups/
```

Minimum object metadata:

```text
import_batch_id
uploaded_by_user_id
source_type
sha256
content_type
original_filename
created_at
```

Original files must be private. Users should access them through short-lived signed URLs generated by the API.

### Tables

#### import_batches

One row per import attempt.

```sql
create table import_batches (
  id uuid primary key,
  import_type text not null check (import_type in (
    'market_products',
    'supplier_products',
    'logistics_rate_cards',
    'orders',
    'fx_rates',
    'cost_templates'
  )),
  status text not null check (status in (
    'uploaded',
    'parsed',
    'validation_failed',
    'ready_for_confirm',
    'confirmed',
    'rejected',
    'rolled_back'
  )),
  source_type text not null,
  source_name text,
  source_url_or_file text,
  confidence_level text check (confidence_level in ('A', 'B', 'C', 'D')),
  original_filename text not null,
  object_storage_key text not null,
  file_sha256 text not null,
  content_type text,
  row_count integer not null default 0,
  valid_row_count integer not null default 0,
  error_count integer not null default 0,
  warning_count integer not null default 0,
  field_mapping_id uuid,
  uploaded_by uuid not null references users(id),
  confirmed_by uuid references users(id),
  uploaded_at timestamptz not null default now(),
  confirmed_at timestamptz,
  rolled_back_at timestamptz,
  notes text
);

create index import_batches_type_status_idx on import_batches(import_type, status);
create index import_batches_uploaded_at_idx on import_batches(uploaded_at desc);
```

#### import_batch_rows

Stores parsed row-level validation results before and after confirmation.

```sql
create table import_batch_rows (
  id uuid primary key,
  import_batch_id uuid not null references import_batches(id) on delete cascade,
  row_number integer not null,
  raw_data jsonb not null,
  mapped_data jsonb,
  validation_status text not null check (validation_status in ('valid', 'warning', 'error')),
  errors jsonb not null default '[]'::jsonb,
  warnings jsonb not null default '[]'::jsonb,
  committed_table text,
  committed_record_id uuid,
  created_at timestamptz not null default now()
);

create unique index import_batch_rows_batch_row_idx on import_batch_rows(import_batch_id, row_number);
create index import_batch_rows_status_idx on import_batch_rows(import_batch_id, validation_status);
```

#### field_mappings

Reusable mapping rules for a source template/vendor/platform.

```sql
create table field_mappings (
  id uuid primary key,
  mapping_type text not null check (mapping_type in (
    'market_products',
    'supplier_products',
    'logistics_rate_cards',
    'orders',
    'fx_rates',
    'cost_templates'
  )),
  name text not null,
  source_name text,
  source_platform text,
  version integer not null default 1,
  status text not null default 'active' check (status in ('draft', 'active', 'archived')),
  mapping_rules jsonb not null,
  required_fields jsonb not null,
  sample_headers jsonb not null default '[]'::jsonb,
  created_by uuid not null references users(id),
  updated_by uuid references users(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create unique index field_mappings_active_name_idx
  on field_mappings(mapping_type, name, version);
```

Example `mapping_rules`:

```json
{
  "平台": "platform",
  "国家": "country",
  "商品名称": "title",
  "售价": "price",
  "币种": "currency",
  "评论数": "reviewCount",
  "销售信号": "salesSignal"
}
```

### Backend Rules

- Validation failures do not write to business tables.
- Warning rows may be confirmed if required fields are valid.
- Confirmed imports create normalized records with `import_batch_id`.
- Rollback should mark records inactive where possible instead of hard deleting.
- Field mapping edits are versioned; do not mutate an old mapping used by a confirmed batch.

## 2. Supplier Matches And Logistics Rate Cards

### Supplier Match Purpose

`supplier_products` are raw/normalized supplier items. `supplier_matches` records the human or automatic relationship between a market product and one or more supplier products.

This allows:

- One market product to have primary and backup suppliers.
- Manual overrides after automatic matching.
- Match confidence and review status.
- Historical changes to supplier strategy.

### Tables

#### supplier_products additions

Existing supplier records should include:

```sql
alter table supplier_products add column import_batch_id uuid references import_batches(id);
alter table supplier_products add column supplier_id_external text;
alter table supplier_products add column stock_status text;
alter table supplier_products add column source_url_or_file text;
alter table supplier_products add column confidence_level text check (confidence_level in ('A', 'B', 'C', 'D'));
alter table supplier_products add column is_active boolean not null default true;
```

#### supplier_matches

```sql
create table supplier_matches (
  id uuid primary key,
  market_product_id uuid not null references market_products(id),
  supplier_product_id uuid not null references supplier_products(id),
  match_type text not null check (match_type in ('primary', 'backup', 'candidate', 'rejected')),
  match_method text not null check (match_method in ('manual', 'title_exact', 'title_fuzzy', 'image', 'import_mapping', 'api')),
  confidence_score numeric(5,2),
  review_status text not null default 'pending' check (review_status in ('pending', 'approved', 'rejected')),
  reason text,
  effective_from timestamptz not null default now(),
  effective_to timestamptz,
  created_by uuid not null references users(id),
  approved_by uuid references users(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index supplier_matches_market_idx on supplier_matches(market_product_id, match_type, review_status);
create index supplier_matches_supplier_idx on supplier_matches(supplier_product_id);
```

Rules:

- Each market product can have only one active approved primary supplier at a time.
- It can have multiple backup suppliers.
- Rejected matches are kept for audit and to avoid repeated bad suggestions.
- Opportunity scoring should use the active approved primary supplier first, then the best approved backup, then candidate matches only as a warning state.

Partial uniqueness for one active primary:

```sql
create unique index supplier_matches_one_primary_idx
  on supplier_matches(market_product_id)
  where match_type = 'primary'
    and review_status = 'approved'
    and effective_to is null;
```

### Logistics Rate Cards

`logistics_rate_cards` must support country, route, weight band, volume weight, effective dates, and quote source.

```sql
create table logistics_rate_cards (
  id uuid primary key,
  provider text not null,
  channel text,
  route text not null,
  origin_country text not null,
  destination_country text not null,
  min_weight_kg numeric(10,3) not null default 0,
  max_weight_kg numeric(10,3),
  volume_weight_kg numeric(10,3),
  price_cny numeric(12,2) not null,
  currency text not null default 'CNY',
  estimated_days_min integer,
  estimated_days_max integer,
  fuel_surcharge_rate numeric(8,4),
  remote_area_fee_cny numeric(12,2),
  effective_from date not null,
  effective_to date,
  status text not null default 'active' check (status in ('draft', 'active', 'expired', 'archived')),
  import_batch_id uuid references import_batches(id),
  source_type text not null,
  source_url_or_file text,
  confidence_level text check (confidence_level in ('A', 'B', 'C', 'D')),
  created_by uuid not null references users(id),
  updated_by uuid references users(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index logistics_rate_cards_lookup_idx
  on logistics_rate_cards(destination_country, origin_country, status, effective_from, effective_to);

create index logistics_rate_cards_provider_idx
  on logistics_rate_cards(provider, route);
```

Rate selection rule:

```text
Given product destination country + estimated package weight:
  1. Filter active cards by origin/destination and effective date.
  2. Match weight band: min_weight_kg <= chargeable_weight <= max_weight_kg.
  3. Prefer lower total cost if service level is acceptable.
  4. Store selected logistics_rate_card_id in cost/profit snapshot.
```

Do not only store `market_products.logistics_cost_cny` long term. Use snapshots:

```sql
create table profit_snapshots (
  id uuid primary key,
  opportunity_id uuid references opportunities(id),
  market_product_id uuid not null references market_products(id),
  supplier_product_id uuid references supplier_products(id),
  supplier_match_id uuid references supplier_matches(id),
  logistics_rate_card_id uuid references logistics_rate_cards(id),
  fx_rate_id uuid,
  cost_template_id uuid,
  sale_price_cny numeric(12,2) not null,
  purchase_price_cny numeric(12,2) not null,
  logistics_cost_cny numeric(12,2) not null,
  landed_cost_cny numeric(12,2) not null,
  gross_profit_cny numeric(12,2) not null,
  gross_margin numeric(8,4) not null,
  created_at timestamptz not null default now(),
  created_by uuid references users(id)
);
```

## 3. Audit Logs, Users, And Roles

### Users And Roles

Minimal internal roles:

| Role | Scope |
| --- | --- |
| `admin` | Full system administration, users, roles, integrations, dangerous config. |
| `ops_lead` | View all operations, approve high-risk operations, configure strategy thresholds. |
| `selection_ops` | Import and edit market products, manage opportunities, submit listing drafts. |
| `supply_chain_ops` | Import suppliers, manage supplier matches, logistics quotes, purchase data. |
| `store_ops` | Manage listing drafts, orders, fulfillment status. |
| `finance_viewer` | Read-only reports, profit, cost, settlement data. |
| `readonly` | Read-only business view. |

#### users

```sql
create table users (
  id uuid primary key,
  email text unique not null,
  display_name text not null,
  password_hash text,
  auth_provider text not null default 'password' check (auth_provider in ('password', 'sso')),
  status text not null default 'active' check (status in ('active', 'disabled', 'invited')),
  last_login_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

#### roles

```sql
create table roles (
  id uuid primary key,
  code text unique not null,
  name text not null,
  description text,
  is_system boolean not null default false,
  created_at timestamptz not null default now()
);
```

#### user_roles

```sql
create table user_roles (
  user_id uuid not null references users(id) on delete cascade,
  role_id uuid not null references roles(id) on delete cascade,
  assigned_by uuid references users(id),
  assigned_at timestamptz not null default now(),
  primary key (user_id, role_id)
);
```

#### permissions

For MVP, permissions can be code-defined by role. If dynamic permission management is needed:

```sql
create table permissions (
  id uuid primary key,
  code text unique not null,
  description text
);

create table role_permissions (
  role_id uuid not null references roles(id) on delete cascade,
  permission_id uuid not null references permissions(id) on delete cascade,
  primary key (role_id, permission_id)
);
```

Recommended permission codes:

```text
import:create
import:confirm
import:rollback
field_mapping:manage
supplier_match:manage
logistics_rate:manage

opportunity:approve
listing:approve

```

### Audit Logs

Every critical action must create an append-only audit record.

```sql
create table audit_logs (
  id uuid primary key,
  actor_user_id uuid references users(id),
  actor_email text,
  action text not null,
  entity_type text not null,
  entity_id uuid,
  request_id text,
  ip_address inet,
  user_agent text,
  before_data jsonb,
  after_data jsonb,
  metadata jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now()
);

create index audit_logs_entity_idx on audit_logs(entity_type, entity_id, created_at desc);
create index audit_logs_actor_idx on audit_logs(actor_user_id, created_at desc);
create index audit_logs_action_idx on audit_logs(action, created_at desc);
```

Actions to audit:

```text
auth.login
auth.logout
import.upload
import.preview
import.confirm
import.rollback
field_mapping.create
field_mapping.update
supplier_match.create
supplier_match.approve
supplier_match.reject
logistics_rate.create
logistics_rate.update
cost_template.update
risk_rule.update
opportunity.approve
listing.approve
listing.publish
order.import
user.create
user.disable
role.assign
```

Audit logging rules:

- Audit logs are append-only.
- Never log raw passwords, API secrets, access tokens, or full buyer personal data.
- For sensitive fields, store masked values or field names only.
- Include `request_id` so API logs and audit rows can be correlated.
- Import confirmation should include `import_batch_id`, row counts, and target tables.

## Implementation Order

1. Add auth, users, roles, and audit log middleware.
2. Add object storage adapter and file upload endpoint.
3. Add `import_batches`, `import_batch_rows`, and `field_mappings`.
4. Move current frontend import normalization into backend import service.
5. Add supplier match persistence and management endpoints.
6. Add logistics rate card persistence and quote selection endpoint.
7. Add rollback for confirmed imports.
8. Add UI migration from `localStorage` to API-backed state.
